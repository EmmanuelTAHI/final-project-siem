## Authentification des utilisateurs

- Mot de passe + code à usage unique envoyé par email (OTP) à chaque
  connexion — voir [Premier login](/docs/getting-started/first-login).
- Détection de "voyage impossible" (connexions géographiquement
  incompatibles en peu de temps).
- Toutes les sessions actives sont révoquées automatiquement lors d'une
  réinitialisation de mot de passe.

## Authentification des agents

Les agents de collecte s'authentifient avec un **token bearer dédié**,
totalement distinct des sessions utilisateur (JWT) :

- Le token est généré côté serveur (256 bits d'entropie), affiché **une
  seule fois** à la création, jamais stocké en clair côté serveur (seul un
  hash SHA-256 est conservé, à la manière d'un mot de passe).
- Ce type de token ne peut **jamais** authentifier une requête vers un
  endpoint utilisateur normal — il n'est reconnu que par l'endpoint
  d'ingestion dédié.
- Un token révoqué est immédiatement invalide pour toute nouvelle requête.

## Chiffrement

- **Au repos** : les identifiants de connecteurs (Microsoft 365, Google
  Workspace, Wazuh) et les tokens OAuth2 sont chiffrés (AES-256/Fernet)
  avant stockage en base de données.
- **En transit** : toutes les communications avec la plateforme SaaS
  passent par HTTPS/TLS (Let's Encrypt, renouvellement automatique). Le mode
  syslog UDP (self-host uniquement, voir [Équipements
  réseau](/docs/agents/network-devices-syslog)) est délibérément exclu de
  cette garantie et documenté comme tel.

## SOC Copilot IA — données envoyées à un fournisseur tiers

Le SOC Copilot (assistant en langage naturel) est **optionnel** : sans clé
`ANTHROPIC_API_KEY` ni `GOOGLE_AI_API_KEY` configurée, il reste désactivé et
n'envoie jamais rien à l'extérieur — le reste de la plateforme fonctionne
normalement.

Quand il est activé, chaque question envoie au fournisseur d'IA configuré
(Anthropic ou Google) : la question de l'analyste, et les résultats
d'outils que le modèle a demandé à exécuter (extraits/agrégats de logs,
d'alertes, d'indicateurs CTI — jamais un accès direct à la base). Ces
résultats sont **toujours filtrés côté serveur par l'organisation de
l'utilisateur authentifié**, jamais par un paramètre fourni par le modèle,
et plafonnés en volume pour éviter toute fuite massive de données brutes.

Points à connaître avant d'activer cette fonctionnalité en présence de
données sensibles :

- Les données transitent par l'infrastructure du fournisseur choisi
  (Anthropic ou Google), soumise à leurs propres conditions de traitement
  des données.
- Aucune donnée n'est envoyée si ni clé Anthropic ni clé Google n'est
  configurée.
- Le mode self-hosted mono-organisation (par défaut) n'a pas de problème
  d'isolation entre clients — seul le choix d'activer ou non l'IA externe
  relève de la politique de confidentialité de l'organisation elle-même.

## Limitation de débit (rate limiting)

Les endpoints sensibles (connexion, inscription, réinitialisation de mot de
passe, code OTP, ingestion d'agents) sont soumis à des limites de débit par
IP/organisation pour limiter l'impact d'un abus ou d'une fuite de
identifiants/token.

## Divulgation responsable

Argus est un projet open source. Si vous découvrez une vulnérabilité de
sécurité, merci de la signaler de façon responsable plutôt que de
l'exploiter ou de la divulguer publiquement avant correction — ouvrez un
ticket privé (Security Advisory) sur le dépôt du projet plutôt qu'une issue
publique.
