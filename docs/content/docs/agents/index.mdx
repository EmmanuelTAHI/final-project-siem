Pour envoyer les logs de vos machines (serveurs, postes de travail,
équipements réseau) vers Log+, vous déployez un **agent** sur chaque
machine à surveiller.

## Agent Log+ natif (recommandé)

Log+ fournit son propre agent — un binaire unique, sans dépendance externe,
pour Linux et Windows. Une PME peut déployer l'ensemble de la chaîne
(plateforme + agents) avec uniquement ce que Log+ fournit lui-même : aucun
logiciel tiers à installer séparément, aucune documentation externe à
suivre.

## 1. Générer un token d'enrôlement

Dans le tableau de bord, allez dans **Agents** → **Générer un token**,
donnez-lui un nom (ex. "Serveurs web prod"). Le token est affiché **une
seule fois** — la commande d'installation complète (Linux ou Windows,
token et URL déjà intégrés) est générée automatiquement à cet instant,
copiez-la immédiatement.

```text
logplus_agt_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

<Callout type="warn">
  Ce token identifie votre organisation de façon unique. Quiconque le
  possède peut envoyer des logs en votre nom — traitez-le comme un mot de
  passe. En cas de doute, révoquez-le immédiatement depuis la page Agents.
</Callout>

## 2. Installer l'agent

Copiez la commande générée par le tableau de bord et exécutez-la sur la
machine à surveiller :

<Tabs items={['Linux', 'Windows']}>
  <Tab value="Linux">
    <Terminal title="bash — root">

    ```bash
    curl -fsSL https://votre-instance.example/agents/install-linux.sh | sudo bash -s -- \
      --url https://votre-instance.example --token logplus_agt_xxxxxxxxxxxxxxxxxxxxxxxxxxxxx
    ```

    </Terminal>

    Installe l'agent comme service **systemd**. [Guide détaillé →](/docs/agents/linux-native)
  </Tab>
  <Tab value="Windows">
    <Terminal title="PowerShell">

    ```powershell
    Invoke-WebRequest -UseBasicParsing -Uri "https://votre-instance.example/agents/install-windows.ps1" -OutFile "$env:TEMP\install-windows.ps1"
    & "$env:TEMP\install-windows.ps1" -Url "https://votre-instance.example" -Token "logplus_agt_xxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
    ```

    </Terminal>

    Installe l'agent comme **service Windows** (élévation demandée automatiquement). [Guide détaillé →](/docs/agents/windows-native)
  </Tab>
</Tabs>

## Méthodes alternatives

Si votre parc utilise déjà un outil de collecte de logs (rsyslog, NXLog,
Fluent Bit), Log+ reste compatible : le point d'entrée HTTP authentifié
ci-dessous fonctionne avec n'importe quel outil qui sait envoyer des
requêtes HTTP.

<Cards>
  <Card title="Linux — rsyslog" href="/docs/agents/linux-rsyslog" />
  <Card title="Windows — NXLog" href="/docs/agents/windows-nxlog" />
  <Card title="Fluent Bit (générique / conteneurs)" href="/docs/agents/fluent-bit" />
  <Card title="Équipements réseau (syslog UDP)" href="/docs/agents/network-devices-syslog" />
</Cards>

Chaque outil doit envoyer une requête :

```
POST https://votre-instance.example/api/ingest/agent/logs/
Authorization: Bearer logplus_agt_xxxxxxxxxxxxxxxxxxxxxxxxxxx
Content-Type: text/plain

<ligne de log 1>
<ligne de log 2>
...
```

Une ligne peut être soit un objet JSON structuré (format utilisé par
l'agent natif — champs `message`, `event_id`, `provider`, `raw_fields`...),
soit une ligne syslog brute (format RFC 3164, `<PRI>message`), soit du texte
libre. Les batchs volumineux peuvent être compressés avec
`Content-Encoding: gzip`.

## 3. Ce qui se passe côté serveur

Au premier envoi, Log+ crée automatiquement un **connecteur** dédié à ce
token, et rattache tous les logs suivants à ce connecteur — jamais à un
connecteur choisi arbitrairement. Vos logs sont ensuite normalisés,
corrélés avec vos propres règles, et analysés par votre propre modèle de
détection d'anomalies (jamais mutualisé avec une autre organisation).

## Limites

- Chaque requête est limitée à 5000 lignes.
- Le débit est limité par organisation (généreux pour un usage normal —
  contactez l'administrateur de l'instance si vous avez un besoin
  particulièrement élevé).
